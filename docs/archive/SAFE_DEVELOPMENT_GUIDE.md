### المبدأ الأساسي

> **"اعتبر كل تغيير في المناطق الحرجة كعملية جراحية دقيقة"**

### المناطق الحرجة في المنصة

| المنطقة | الملفات | مستوى الخطورة |
|---------|---------|---------------|
| **المصادقة** | `AuthContext.tsx`, `ProtectedRoute.tsx` | 🔴 حرج |

## ⚡ فحص الأداء

### مؤشرات الأداء الرئيسية (KPIs)

| المؤشر | الهدف | الحد الأقصى | الحالة الحالية |
|--------|-------|-------------|----------------|
| **LCP** | < 2.5s | 4s | ✅ < 2.5s |
| **FCP** | < 1.8s | 3s | ✅ < 1.8s |
| **CLS** | < 0.1 | 0.25 | ✅ < 0.1 |
| **TTFB** | < 0.8s | 1.8s | ✅ < 0.8s |
| **INP** | < 200ms | 500ms | ✅ < 200ms |

### قائمة فحص الأداء

#### 1. استدعاءات قاعدة البيانات والشبكة

```typescript
// ❌ سيئ - استدعاءات متكررة غير ضرورية
useEffect(() => {
  fetchData(); // يُستدعى في كل render
});

// ✅ جيد - استدعاء محدد
useEffect(() => {
  fetchData();
}, []); // يُستدعى مرة واحدة

// ✅ أفضل - استخدام React Query
const { data } = useQuery({
  queryKey: ['data'],
  queryFn: fetchData,
  staleTime: 5 * 60 * 1000, // 5 دقائق
});
```

#### 2. إعادة الـ Renders غير الضرورية

```typescript
// ❌ سيئ - إعادة render في كل تغيير
function TaskItem({ task }) {
  return <div>{task.name}</div>;
}

// ✅ جيد - استخدام memo
const TaskItem = React.memo(({ task }) => {
  return <div>{task.name}</div>;
});
```

#### 3. تحسين الصور والأصول

```typescript
// ❌ سيئ - صور ضخمة بدون تحسين
<img src="/large-image.jpg" />

// ✅ جيد - تحميل كسول مع placeholder
import { OptimizedImage } from '@/components/shared/PerformanceOptimizer';

<OptimizedImage 
  src="/optimized-image.webp" 
  alt="وصف" 
  priority={false}
/>
```

#### 4. Code Splitting

```typescript
// ❌ سيئ - تحميل كل شيء مرة واحدة
import HeavyComponent from '@/components/HeavyComponent';

// ✅ جيد - تحميل كسول
const HeavyComponent = lazyWithRetry(() => 
  import('@/components/HeavyComponent')
);
```

### أوامر فحص الأداء

```bash
# فحص Web Vitals محلياً
npm run lighthouse

# فحص حجم الحزمة
npm run build
du -sh dist/

# تحليل الحزمة
npm run build -- --analyze
```

## 🔧 تدفقات التصحيح النموذجية

### التدفق 1: "علقت في حلقة أخطاء"

```
1. توقف ولا تحاول المزيد من الإصلاحات
           ↓
2. اجمع المعلومات:
   - ما الخطأ بالضبط؟
   - ما الذي جربته؟
   - متى بدأت المشكلة؟
           ↓
3. استخدم Chat Mode (بدون تغيير كود)
   - ناقش المشكلة
   - حلل الأسباب المحتملة
           ↓
4. قسّم المشكلة لأجزاء صغيرة
   - اختبر كل جزء منفصلاً
           ↓
5. ابدأ من مكان يعمل
   - استخدم History للرجوع
   - ابنِ تدريجياً
```

### التدفق 2: "المكون لا يعمل أبداً"

```
1. هل المكون موجود في DOM؟
   → DevTools → Elements
           ↓ نعم
2. هل يستلم Props صحيحة؟
   → React DevTools → Props
           ↓ نعم
3. هل الـ State صحيحة؟
   → React DevTools → State
           ↓ نعم
4. هل هناك شرط يمنع الـ Render؟
   → راجع conditional rendering
           ↓ لا
5. هل هناك خطأ CSS؟
   → visibility: hidden?
   → display: none?
   → opacity: 0?
```

### التدفق 3: "الـ API لا تستجيب"

```
1. هل الاتصال بالإنترنت يعمل؟
   → window.waqfDebug.healthStatus()
           ↓ نعم
2. هل الـ Request وصل للخادم؟
   → Network tab → Status
           ↓ نعم (4xx/5xx)
3. ما نوع الخطأ؟
   → 401: مشكلة مصادقة
   → 403: مشكلة صلاحيات
   → 404: المسار خاطئ
   → 500: خطأ في الخادم
           ↓
4. حل حسب النوع
```

## 🔄 التدفقات التفاعلية مع AI

هذا القسم يوضح كيفية التفاعل بفعالية مع AI لحل المشكلات الشائعة.

### السيناريو 1: "علقت في حلقة أخطاء" (Error Loop)

**الموقف:** طلبت ميزة معقدة، التطبيق لا يبني، و"Try to Fix" فشل مرتين أو أكثر.

**المبدأ الأساسي:** وصف الخطأ بدقة وتأكيد فهم AI قبل الإصلاح - لا تضغط "Try to Fix" بشكل أعمى.

```
│         │                                                                   │
│         ▼                                                                   │

│                         │                                                   │
```

**المبدأ الأساسي:** صف ما توقعته وما حصل فعلاً، مع مشاركة مقتطفات السجلات إن وُجدت.

### السيناريو 3: "مكون UI اختفى" (Dead Component)

**الموقف:** بعد إعادة هيكلة، قسم كامل من الواجهة اختفى فجأة.

**المبدأ الأساسي:** التحقق من وجود المكون في DOM أولاً، ثم تتبع سلسلة الـ render.


### جدول الأسئلة الذهبية

| السؤال | متى تسأله | ما تتوقعه من AI |
|--------|-----------|-----------------|
| **"ما السبب الجذري لهذا الخطأ؟"** | عند أي خطأ بناء أو runtime | تحليل تقني للسبب الحقيقي |

| **"أين يُستدعى هذا المكون؟"** | عند اختفاء مكون | قائمة بأماكن الاستدعاء |
| **"ما الـ state الحالية لهذا المكون؟"** | عند سلوك غير متوقع | عرض state و props |

### Agent Mode (الوضع الافتراضي)

**القدرات الكاملة:**
- ✅ تفسير الطلب وفهم السياق تلقائياً
- ✅ استكشاف الكود والملفات عند الحاجة
- ✅ قراءة الملفات وتحليل التبعيات
- ✅ إصلاح وإعادة هيكلة الكود
- ✅ تصحيح المشاكل بشكل استباقي
- ✅ فحص سجلات Console والشبكة تلقائياً
- ✅ البحث في الويب للحصول على توثيق
- ✅ توليد وتعديل الصور للتطبيق

**التسعير:** حسب تعقيد الرسالة
- رسائل بسيطة: أقل من 1 credit
- رسائل معقدة: أكثر من 1 credit

### Chat Mode (وضع المحادثة)

**القدرات:**
- ✅ مناقشة المشاكل والتخطيط
- ✅ تصحيح الأخطاء بالتحليل العميق
- ✅ البحث في الملفات والكود
- ✅ فحص السجلات وقاعدة البيانات
- ❌ **لا يمكنه تغيير الكود**

**التسعير:** 1 credit لكل رسالة

**أفضل استخدامات Chat Mode:**
1. **تصحيح فعال:** طلب التحقيق العميق قبل الإصلاح
2. **تخطيط قواعد البيانات:** تصميم الهيكل والعلاقات
3. **تحديد التغييرات:** معرفة الحد الأدنى للتغييرات
4. **طلب نصائح:** تحسين الكود والأداء
5. **تقييم التأثير:** فهم تأثير الميزات الجديدة

## 🖥️ استخدام أدوات المتصفح وسجلات Console

### التعامل مع أخطاء JavaScript المضغوطة

**مثال على خطأ مضغوط:**
```
TypeError: Q9() is undefined
at https://example.lovable.app/assets/index-DWQbrtrQQj.js:435:39117
```

**لماذا الكود مضغوط؟**
- في بيئة الإنتاج، الكود يُضغط ويُشفّر لتحسين الأداء
- أسماء المتغيرات والدوال تُختصر (مثل `Q9`, `Xa`, `_3`)
- الأسطر تُدمج والمسافات تُحذف

### كيف تتعامل مع الأخطاء المضغوطة

| الخطوة | الإجراء | التفاصيل |
|--------|---------|----------|
| 1️⃣ | انسخ الخطأ كاملاً | شامل stack trace كاملاً |
| 2️⃣ | صف المشكلة | "التطبيق لا يعمل والشاشة فارغة" |
| 3️⃣ | شارك مع AI | AI يحلل رغم ضغط الكود |
| 4️⃣ | اطلب الحل | AI يحدد الملف والسطر الأصلي |

### أنواع أخطاء Console الشائعة

| نوع الخطأ | المعنى | الحل السريع |
|-----------|--------|-------------|
| `TypeError: X is undefined` | محاولة استخدام قيمة غير معرفة | فحص تهيئة المتغير |
| `TypeError: X is not a function` | محاولة استدعاء شيء ليس دالة | تحقق من الاستيراد |
| `SyntaxError` | خطأ في صياغة الكود | مراجعة الملف المذكور |
| `NetworkError` | فشل في الاتصال | فحص API والاتصال |
| `ChunkLoadError` | فشل تحميل جزء من التطبيق | مسح cache وإعادة تحميل |
| `CORS error` | مشكلة في Cross-Origin | تحقق من إعدادات الخادم |

## 🔍 تحليل السبب الجذري مقابل الأعراض

### المبدأ الأساسي

> **"اسأل دائماً 'لماذا حدث هذا؟' وليس فقط 'ماذا أفعل الآن؟'"**

### الفرق بين الأعراض والسبب الجذري

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   🔴 العَرَض (Symptom)              🟢 السبب الجذري (Root Cause)   │
│   ─────────────────                ──────────────────────────   │
│                                                                 │
│   "user = null"                    "API فشل بصمت بدون handling" │
│        │                                    │                   │
│        ▼                                    ▼                   │
│   ❌ إضافة:                         ✅ إضافة:                   │
│   if (user !== null)               try/catch + error toast     │
│        │                                    │                   │
│        ▼                                    ▼                   │
│   المشكلة ستتكرر                    المشكلة تُحل نهائياً         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### مثال عملي

**❌ معالجة أعراض (سيئ):**
```typescript
// المشكلة: user أحياناً null
// الحل السريع: فحص null في كل مكان
if (user !== null) {
  displayUserName(user.name);
}
// ⚠️ لكن لماذا كان user = null أصلاً؟
```

**✅ معالجة سبب جذري (جيد):**
```typescript
// التحقيق: user كان null لأن الجلب فشل بصمت
// الحل: إضافة error handling للـ API

async function loadUser() {
  try { 
    const user = await fetchUser(); 
    displayUserName(user.name);
  } catch (error) { 
    showErrorToast("فشل تحميل بيانات المستخدم");
    console.error("User fetch failed:", error);
  }
}
```

### أسئلة تحليل السبب الجذري

استخدم هذه الأسئلة مع AI:

1. **"أرى أنك أصلحت الخطأ بإضافة فحص، لكن لماذا كانت القيمة null أصلاً؟"**

2. **"هل يمكننا معالجة السبب بدلاً من الأعراض؟"**

3. **"ما العلاقة السببية بين X و Y؟"**

4. **"إذا أصلحنا هذا، هل ستظهر المشكلة في مكان آخر؟"**

## ↩️ الرجوع بحكمة (Rollback Wisely)

### متى تستخدم الرجوع؟

| الموقف | القرار | السبب |
|--------|--------|-------|
| الكود متشابك بعد عدة إصلاحات فاشلة | ✅ **ارجع** | أسهل من فك التشابك |
| تغيير صغير كسر شيء كبير | ✅ **ارجع** | توفير الوقت |
| غير متأكد مما تغير | ✅ **ارجع** | نقطة بداية نظيفة |
| المشكلة واضحة ومحددة | ❌ أصلح | أسرع من الرجوع |
| التغيير بسيط وموضعي | ❌ أصلح | لا داعي للرجوع |

### كيفية الرجوع الفعال

```
┌────────────────────────────────────────────────────────────┐
│                     خطوات الرجوع الفعال                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   1️⃣ افتح History في Lovable                               │
│              ↓                                             │
│   2️⃣ حدد النقطة الآمنة (آخر إصدار يعمل)                    │
│              ↓                                             │
│   3️⃣ اضغط "Restore" على تلك النقطة                        │
│              ↓                                             │
│   4️⃣ أعلم AI بما فعلت (القالب أدناه)                       │
│              ↓                                             │
│   5️⃣ ابدأ التطبيق التدريجي بخطوات صغيرة                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 📈 التحسين التدريجي (Progressive Enhancement)

### المبدأ الأساسي

> **"ابنِ الميزات في خطوات صغيرة قابلة للاختبار"**

### مقارنة: طلب واحد كبير vs طلبات متدرجة

**❌ سيئ: طلب واحد كبير**
```
"أريد نظام إشعارات كامل مع:
- إشعارات داخلية
- إرسال email
- إرسال SMS
- push notifications
- لوحة تحكم للإشعارات
- إعدادات تفضيلات المستخدم
- جدولة الإشعارات
..."

⚠️ احتمال الفشل عالي
⚠️ صعوبة تحديد مصدر الخطأ
⚠️ وقت طويل للتنفيذ
```

**✅ جيد: طلبات متدرجة**
```
Prompt 1: "أنشئ جدول notifications في قاعدة البيانات"
  → اختبار ✅

Prompt 2: "أضف دالة createNotification"
  → اختبار ✅

Prompt 3: "أضف مكون NotificationBell في الـ navbar"
  → اختبار ✅

Prompt 4: "أضف صفحة قائمة الإشعارات"
  → اختبار ✅

Prompt 5: "أضف إرسال email عند الإشعارات المهمة"
  → اختبار ✅

...وهكذا
```

### فوائد التدرج

| الفائدة | الشرح |
|---------|-------|
| **تصحيح سهل** | تعرف بالضبط أي خطوة سببت المشكلة |
| **مرونة عالية** | يمكنك تغيير الاتجاه بسهولة |
| **فهم أفضل** | تفهم كل جزء قبل الانتقال للتالي |
| **جودة أعلى** | كل جزء مختبر جيداً |
| **توفير الوقت** | أخطاء أقل = وقت أقل للتصحيح |

## 📝 التوثيق أثناء العمل

### لماذا التوثيق مهم؟

> **"ما لا يُوثق، يُنسى"**

بعد حل مشكلة صعبة، ستنسى التفاصيل بعد أسبوع. التوثيق يحفظ المعرفة.

### طلب ملخص من AI

بعد حل مشكلة معقدة، اطلب:

```
"لخص ما كانت المشكلة وكيف حللناها"
```

### مثال على ملخص AI

```markdown
## ملخص المشكلة والحل

### المشكلة
الإشعارات لم تكن ترسل رغم عدم ظهور أخطاء في Console.

### التحقيق
1. فحصنا Network tab - الطلبات تُرسل بنجاح (200 OK)
2. فحصنا Edge Function logs - لا توجد أخطاء
3. فحصنا خدمة البريد - **المفتاح منتهي الصلاحية!**

### السبب الجذري
مفتاح API لخدمة Resend كان منتهي الصلاحية منذ 3 أيام.

### الحل
1. تحديث مفتاح API في Supabase Secrets
2. إضافة logging لتسهيل اكتشاف مشاكل مشابهة
3. إضافة تنبيه عند فشل إرسال البريد

### الدروس المستفادة
- فحص صلاحية مفاتيح API بشكل دوري
- إضافة monitoring لخدمات الطرف الثالث
```
